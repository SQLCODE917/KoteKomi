# KoteKomi

## Design Document v0.3

## 1. Context & Problem

### 1.1 Glossary

**KoteKomi**
The project. A local-first system that tracks public-world intelligence as structured records.

**Actor**
A person or role-bearing entity.

**Organization**
A company, agency, lab, publisher, university, nonprofit, think tank, or other institution.

**Event**
A bounded real-world happening with time, participants, and optional place.

**Source**
An external artifact that contains information. Examples: article, transcript, PDF, blog post, social post, filing, press release.

**Document**
A local archived copy of a Source or extracted content derived from a Source.

**EvidenceSpan**
A precise pointer to evidence inside a Document.

**Assertion**
An atomic statement about the world, a Source, another Assertion, or an analytic inference.

**Relationship**
A typed edge between two domain objects. A Relationship is derived from one or more accepted Assertions.

**Outcome**
A real-world result linked to Actors, Organizations, Events, or Assertions.

**ArgumentEdge**
A typed edge between two Assertions. It records support, contradiction, weakening, contextualization, inference, or corroboration.

**ProvenanceActivity**
A recorded action that created or changed domain records. Examples: source fetch, text extraction, model extraction, human review, briefing generation.

**ProposedChange**
A model-generated or script-generated change that awaits human review before it changes accepted state.

**Briefing**
A generated report that explains changed state since a previous Briefing.

**Ledger**
The canonical local SQLite database that stores accepted domain records, ProposedChange records, and ProvenanceActivity records.

**Archive**
The local file tree that stores raw Sources, extracted text, attachments, Briefings, and exports.

**Domain Core**
The package that defines domain types, domain rules, and ontology semantics. The Domain Core has no dependency on tools, storage engines, model runtimes, or user interfaces.

**Application Layer**
The package that implements use cases through ports. Examples: add Source, extract Assertions, review ProposedChange, mine Relationships, generate Briefing.

**Adapter**
A package that implements a port for a specific tool. Examples: SQLite adapter, LanceDB adapter, Ollama adapter, llama.cpp adapter, NetworkX adapter.

**Port**
An interface that the Application Layer calls. A Port hides a tool-specific Adapter behind a stable contract.

**Pipeline**
A command-line workflow that composes Application Layer use cases.

**Ontology Profile**
The project’s internal ontology, based on selected external standards and reduced to the objects this project uses.

### 1.2 Problem

A daily news summary loses state. It reports new items but does not maintain durable knowledge about Actors, Organizations, Events, Sources, Assertions, EvidenceSpans, Outcomes, and Relationships.

KoteKomi solves a different problem: it preserves an intelligence state and generates a Briefing from changes in that state.

The first target domain is frontier AI model release governance. The system tracks claims about government review, institutional pressure, safety evaluations, delayed releases, negotiated clearances, public statements, and related Actors or Organizations.

The Domain Core remains general so the system can support other public-world intelligence topics later.

### 1.3 Example question

The system should support questions shaped like this:

> Person A and Person B were reported to have negotiated the release of Model X last week. This week, Person A appeared at Event Y. Event Y also included Person C. Person C was involved in an earlier model-release review. What changed, what is supported by evidence, and what remains inference?

The system answers by inspecting Assertions, EvidenceSpans, Relationships, ArgumentEdges, Outcomes, and prior Briefings.

## 2. Product Goal

KoteKomi builds a local-first intelligence ledger and briefing generator.

The system tracks:

* Actors
* Organizations
* Events
* Sources
* Documents
* EvidenceSpans
* Assertions
* Relationships
* ArgumentEdges
* Outcomes
* ProvenanceActivities
* ProposedChanges
* Briefings

The system produces a daily Briefing that answers:

* Which accepted Assertions changed?
* Which ProposedChanges require review?
* Which new Actors, Organizations, Events, Sources, or Outcomes appeared?
* Which Relationships changed?
* Which ArgumentEdges changed the interpretation of an Assertion?
* Which prior Briefing assessment now needs revision?
* Which Assertions gained or lost confidence?

## 4. System Style

The system follows a Unix-style pipeline.

Each Pipeline step has one responsibility and a stable input/output contract.

```text
fetch_source
→ extract_document_text
→ create_document_chunks
→ embed_document_chunks
→ propose_assertions
→ reconcile_entities
→ review_proposed_changes
→ update_relationships
→ mine_connections
→ generate_briefing
→ export_records
```

Each step reads and writes one or more of:

* JSON
* JSONL
* SQLite rows
* local files
* Markdown

The Ledger stores canonical state. Derived stores can be rebuilt.

Derived stores include:

* vector index
* graph projection
* Markdown Briefing files
* JSONL exports
* GraphML exports
* Mermaid exports

## 5. Architecture

### 5.1 Layers

```text
Domain Core
  types, value objects, ontology rules, validation rules

Application Layer
  use cases and ports

Adapters
  SQLite, file storage, model runtimes, vector index, graph analysis, web fetchers

Pipelines
  command-line workflows that compose use cases

Presentation
  Markdown Briefings, CLI output, future UI
```

### 5.2 Dependency rule

Dependencies point inward.

```text
Pipelines → Application Layer → Domain Core
Adapters → Application Layer → Domain Core
Domain Core → no external package
```

### 5.3 Domain Core boundary

The Domain Core defines:

* domain objects
* controlled vocabularies
* validation rules
* status transitions
* confidence dimensions
* provenance semantics
* ontology mappings

The Domain Core does not import:

* SQLite libraries
* LanceDB libraries
* NetworkX
* Ollama clients
* llama.cpp clients
* vLLM clients
* web scraping libraries
* Markdown renderers
* RDF libraries

### 5.4 Port and Adapter boundary

The Application Layer defines Ports. Adapters implement Ports.

| Port               | MVP Adapter                    | Later Adapter               |
| ------------------ | ------------------------------ | --------------------------- |
| `LedgerRepository` | SQLite                         | Postgres                    |
| `ArchiveStore`     | local filesystem               | object storage              |
| `VectorIndex`      | LanceDB or sqlite-vec          | Qdrant                      |
| `ModelRuntime`     | Ollama or llama.cpp            | MLX, vLLM, cloud LLM        |
| `GraphAnalyzer`    | NetworkX                       | Neo4j                       |
| `SourceFetcher`    | trafilatura/manual file import | browser capture, search API |
| `BriefingWriter`   | Markdown file writer           | web UI, email, task output  |

### 5.5 Boundary validation

Domain Core records and Application Layer DTOs define structured boundary contracts.

Every Port, Adapter, and Pipeline boundary parses inbound structured values into the declared Domain Core record or Application Layer DTO before passing them inward.

Every boundary serializes outbound structured values from the declared Domain Core record or Application Layer DTO before passing them outward.

Deterministic project-owned boundaries fail fast when parsing or validation fails.

Deterministic project-owned boundaries do not drop, repair, skip, coerce, or clean up invalid values.

Accepted Ledger writes validate Domain Core record shape before commit.

Accepted Ledger writes also validate cross-record references before commit.

The Domain Core validates intrinsic record rules.

The Application Layer validates Ledger references that require repository state.

Model output is non-deterministic.

Model output can enter explicit recovery paths such as rejection, quarantine, or reviewable ProposedChange records.

Model output never writes accepted state directly.

### 5.6 Code quality commitments

Application Layer use cases own domain decisions.

Adapters translate, validate, persist, and load records.

Adapters do not decide Domain meaning, status transitions, review outcomes, or repair policy.

Public Application Layer use cases use explicit input and result DTOs.

Each use case defines its atomic read, validate, write, and provenance unit.

Pipeline commands open Ledger transactions around use cases.

Every accepted Ledger state change creates or references a ProvenanceActivity.

Required accepted-state invariants live in Domain Core validation or Application Layer reference validation.

Record-type dispatch covers every supported Domain Core record type explicitly or fails.

Nontrivial boundary mappings use named mapping functions.

Outbound structured values come from Domain Core records or Application Layer DTOs.

The Ledger and Archive store canonical state.

Graph projections, vector indexes, Briefings, and exports are derived state.

Derived state rebuilds from the Ledger and Archive.

Happy-path fixtures are internally coherent.

Malformed fixtures are named as negative fixtures and assert the exact failure.

## 8. MVP User Stories

### 8.1 Add Source

As a user, I add a URL or local file.

The system creates:

* Source
* Document
* ProvenanceActivity

Acceptance criteria:

* `Source.id` is stable.
* `Document.id` is stable.
* raw content is stored in the Archive.
* extracted text is stored in the Archive.
* the Ledger records the ProvenanceActivity that created the Source and Document.

### 8.2 Propose Assertions

As a user, I run a Pipeline that proposes Assertions from a Document.

The system creates:

* ProposedChange records
* EvidenceSpan records when the model identifies exact support text
* ProvenanceActivity for the model run

Acceptance criteria:

* each ProposedChange references one Document.
* each ProposedChange records model name and prompt ID.
* each ProposedChange stores proposed JSON.
* each ProposedChange has `review_status = pending`.

### 8.3 Review ProposedChange

As a user, I approve, reject, or edit a ProposedChange.

The system creates or updates:

* Assertion
* EvidenceSpan
* Relationship
* ArgumentEdge
* ProvenanceActivity

Acceptance criteria:

* approved ProposedChange records create accepted records.
* rejected ProposedChange records keep their proposed JSON.
* edited ProposedChange records keep the original proposed JSON and the edited accepted JSON.
* the Ledger records the reviewing Agent.

### 8.4 Track Assertion

As a user, I track an Assertion over time.

The system updates:

* Assertion status
* confidence dimensions
* ArgumentEdges
* EvidenceSpans
* current assessment

Acceptance criteria:

* each Assertion status change creates a ProvenanceActivity.
* each confidence change creates a ProvenanceActivity.
* each new supporting or contradicting Assertion creates an ArgumentEdge.
* prior assessment text remains available through revision history.

### 8.5 Mine Relationships

As a user, I run connection mining.

The system reads accepted Assertions and proposes Relationships.

Acceptance criteria:

* each proposed Relationship references the Assertions that support it.
* each inferred Relationship has `assertion_type = analytic_inference`.
* each inferred Relationship has `world_truth_confidence` and `causal_confidence` when causal language appears.
* each inferred Relationship enters the Ledger through ProposedChange review.

### 8.6 Generate Briefing

As a user, I generate a daily Briefing.

The system creates:

* Briefing record
* Markdown Briefing file
* structured citation registry file
* ProvenanceActivity

Acceptance criteria:

* the Briefing compares current state with the previous Briefing.
* the Briefing lists changed Assertions.
* the Briefing lists changed Relationships.
* the Briefing lists changed ArgumentEdges.
* the Briefing lists new Sources.
* the Briefing labels each analytic inference as an inference.
* the Briefing Markdown uses numbered citations for source-backed claims.
* the Briefing citation registry stores Source and EvidenceSpan IDs.
* the Markdown Briefing uses compact citation numbers that resolve to the structured citation registry.
* agents resolve Briefing citations through structured registry data, not by parsing Markdown.
* default human-facing Briefing Markdown does not expose raw canonical Domain IDs.
* the Briefing includes a deterministic Bottom Line, Key Judgments, Evidence Basis, Uncertainties and Gaps, and Analytic Trace.
* the Briefing narrative preserves evidence boundaries and does not introduce unsupported claims.

### 8.7 Run on both target machines

As a user, I run the same project on macOS and WSL/Ubuntu.

Acceptance criteria:

* Pipelines run through documented commands.
* runtime-specific configuration lives outside the Domain Core.
* the Ledger remains SQLite-compatible.
* local paths come from configuration.
* model runtime selection comes from configuration.
* the default `macbook` runtime profile uses llama.cpp server with
  `hf.co/Qwen/Qwen3-14B-GGUF:Q4_K_M` at a 16,384-token context window.
* the explicit `wsl-4090` runtime profile uses Ollama with `qwen3:30b` at a
  16,384-token context window.

## 9. Ontology Profile

### 9.1 Profile name

```text
KoteKomi Ontology Profile v0.1
```

### 9.2 Profile purpose

The Ontology Profile defines the Domain Core objects and maps them to established ontology patterns.

The project uses the profile internally as plain domain objects. Exporters can map those objects to external standards later.

### 9.3 Source standards

| Standard pattern         | Project use                                                               |
| ------------------------ | ------------------------------------------------------------------------- |
| Wikidata statement model | Assertion subject-predicate-object pattern with qualifiers and references |
| W3C PROV-O               | ProvenanceActivity model                                                  |
| Nanopublication pattern  | Assertion + provenance + publication metadata shape                       |
| W3C Web Annotation       | EvidenceSpan targeting                                                    |
| Simple Event Model       | Event, Actor, Place, Time pattern                                         |
| CIDOC CRM                | future reference for deeper event modeling                                |
| AIF                      | ArgumentEdge semantics                                                    |
| Schema.org / ClaimReview | export format                                                             |

### 9.4 Core rule

The Ontology Profile centers on Assertion.

The system uses Assertion rather than Fact because the system tracks uncertainty, contradiction, and inference.

### 9.5 Assertion shape

```text
Assertion
  id
  assertion_type
  epistemic_scope
  subject_entity_id
  predicate
  object_entity_id
  object_value
  status
  source_authority
  attribution_basis
  attributed_to_id
  source_report_confidence
  extraction_confidence
  world_truth_confidence
  causal_confidence
  qualifiers
  current_assessment
  authority_source_ids
  authority_evidence_span_ids
  created_at
  updated_at
```

### 9.6 Assertion types

| Assertion type         | Meaning                                             |
| ---------------------- | --------------------------------------------------- |
| `reported_observation` | A Source reports that something happened.           |
| `source_claim`         | A Source makes or repeats a claim.                  |
| `direct_quote`         | A Source quotes an Actor.                           |
| `analytic_inference`   | The system or user infers a connection.             |
| `corroboration`        | One Assertion supports another Assertion.           |
| `contradiction`        | One Assertion conflicts with another Assertion.     |
| `outcome_observation`  | A Source reports a real-world result.               |
| `status_update`        | The system records a change to an Assertion status. |

### 9.7 Assertion statuses

| Status         | Meaning                                                                            |
| -------------- | ---------------------------------------------------------------------------------- |
| `proposed`     | A model or script proposed the Assertion.                                          |
| `reported`     | A Source reports the Assertion.                                                    |
| `corroborated` | Multiple independent EvidenceSpans support the Assertion.                          |
| `confirmed`    | An authoritative Source or direct primary evidence supports the Assertion.         |
| `contradicted` | Evidence conflicts with the Assertion.                                             |
| `superseded`   | A newer Assertion replaces this Assertion.                                         |
| `retracted`    | The original Source or Actor withdrew the Assertion.                               |
| `deprecated`   | The system keeps the Assertion for history but excludes it from current synthesis. |

### 9.8 Confidence dimensions

| Dimension                  | Meaning                                                       |
| -------------------------- | ------------------------------------------------------------- |
| `source_report_confidence` | Confidence that the Source said the thing.                    |
| `extraction_confidence`    | Confidence that the system parsed the Source correctly.       |
| `world_truth_confidence`   | Confidence that the Assertion is true.                        |
| `causal_confidence`        | Confidence that one thing caused or influenced another thing. |

### 9.9 Epistemic scope and Source authority

| Epistemic scope        | Meaning                                                    |
| ---------------------- | ---------------------------------------------------------- |
| `source_report`        | The Assertion states what a Source reports.                |
| `attributed_statement` | The Assertion states what an Actor or Organization said.   |
| `world_state`          | The Assertion states a world condition or event.           |
| `causal_explanation`   | The Assertion states a causal explanation.                 |
| `analytic_inference`   | The Assertion states a KoteKomi inference.                 |

| Source authority | Meaning                                                   |
| ---------------- | --------------------------------------------------------- |
| `primary`        | The Assertion references direct primary evidence.         |
| `secondary`      | The Assertion references reporting or analysis.           |
| `tertiary`       | The Assertion references aggregated or summary material.  |
| `unknown`        | The Source authority cannot be determined from the Source. |
| `not_applicable` | The Assertion is not Source-backed.                       |

| Attribution basis   | Meaning                                                    |
| ------------------- | ---------------------------------------------------------- |
| `direct_document`   | The EvidenceSpan is from the issuing Source itself.        |
| `quoted_statement`  | The EvidenceSpan quotes the attributed Actor or Organization. |
| `reported_by_source` | The EvidenceSpan reports attribution without direct source text. |
| `anonymous_source`  | The EvidenceSpan attributes the claim to unnamed people or materials. |
| `unclear`           | The EvidenceSpan does not establish attribution clearly.   |
| `not_applicable`    | The Assertion has no attribution basis.                    |

### 9.10 EvidenceSpan shape

```text
EvidenceSpan
  id
  source_id
  document_id
  assertion_id
  selector_type
  exact_text
  prefix_text
  suffix_text
  location
  created_at
```

### 9.11 ArgumentEdge shape

```text
ArgumentEdge
  id
  from_assertion_id
  to_assertion_id
  relation
  rationale
  evidence_span_ids
  confidence
  created_at
```

### 9.12 ArgumentEdge relations

| Relation         | Meaning                                                                            |
| ---------------- | ---------------------------------------------------------------------------------- |
| `supports`       | The source Assertion increases confidence in the target Assertion.                 |
| `contradicts`    | The source Assertion decreases confidence in the target Assertion by conflict.     |
| `weakens`        | The source Assertion reduces confidence without direct contradiction.              |
| `contextualizes` | The source Assertion changes interpretation without changing confidence by itself. |
| `infers`         | The source Assertion provides a premise for an analytic inference.                 |
| `corroborates`   | The source Assertion independently supports the target Assertion.                  |

## 10. Domain Rules

The Domain Core enforces these rules.

### 10.1 Assertion rules

1. An Assertion has one subject.
2. An Assertion has one predicate.
3. An Assertion has either one object entity or one object value.
4. An Assertion has an EpistemicScope.
5. An Assertion has a SourceAuthority.
6. An Assertion has an AttributionBasis.
7. An accepted Assertion has at least one ProvenanceActivity.
8. A Source-backed Assertion has at least one Source.
9. A Source-backed accepted Assertion has at least one EvidenceSpan.
10. An analytic inference has `assertion_type = analytic_inference`.
11. An analytic inference has `epistemic_scope = analytic_inference`.
12. A non-source analytic inference has `source_authority = not_applicable`.
13. A non-source analytic inference has `attribution_basis = not_applicable`.
14. An attributed statement has `attributed_to_id`.
15. A primary-source-backed Assertion has authority Source IDs and authority EvidenceSpan IDs.
16. A causal analytic inference has `causal_confidence`.
17. A contradiction creates an ArgumentEdge with `relation = contradicts`.
18. A superseded Assertion keeps its original record.

### 10.2 ProposedChange rules

1. A ProposedChange starts with `review_status = pending`.
2. A reviewer can approve, reject, or edit a ProposedChange.
3. An approved ProposedChange creates or updates accepted records.
4. A rejected ProposedChange remains in the Ledger.
5. An edited ProposedChange stores both original proposed JSON and accepted JSON.
6. Each review action creates a ProvenanceActivity.

### 10.3 Briefing rules

1. A Briefing compares current Ledger state with the previous Briefing.
2. A Briefing Markdown uses numbered citations for Source-backed Assertions.
3. A Briefing citation registry stores Source IDs and EvidenceSpan IDs for accepted Source-backed Assertions.
4. A Briefing stores numbered citations as structured derived data alongside the Markdown file.
5. Agents resolve Briefing citations through the structured citation registry, not by parsing Markdown.
6. Default human-facing Briefing Markdown does not expose raw canonical Domain IDs.
7. A Briefing labels analytic inferences.
8. A Briefing records the ProvenanceActivity that generated it.
9. A Briefing narrative is derived from accepted Ledger records.
10. A Briefing narrative preserves Source, EvidenceSpan, Assertion, Relationship, ArgumentEdge, and Outcome boundaries.
